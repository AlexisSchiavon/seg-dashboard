<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Public internet -> /health | Unauthenticated liveness probe; must not leak internal detail |
| Browser cookie -> /auth/change-password, /auth/users | Account mutations require a verified auth cookie |
| Browser fetch -> protected routes | 401 handling drives session-expiry UX |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | /health response | mitigate | Body limited to `{status, database}` coarse strings; no stack traces, versions, or connection strings; always 200 (no error-detail leakage via 5xx) |
| T-03-02 | Elevation of Privilege | /auth/change-password | mitigate | Re-verify current_password before accepting new_password; guarded by get_current_user (cannot change another user's password — operates on current_user only) |
| T-03-03 | Spoofing / EoP | /auth/users | mitigate | Guarded by get_current_user; single-tier auth means only the authenticated admin can create users (FUT-03 RBAC deferred); password hashed via pwdlib before storage |
| T-03-04 | Input Validation | change-password / users bodies | mitigate | Pydantic schemas (ChangePasswordRequest, UserCreate with EmailStr) auto-validate; 422 on invalid input |
| T-03-05 | Information Disclosure | duplicate-user creation | accept | 409 on existing email reveals existence, but endpoint is auth-gated (single internal admin) — acceptable in v1 |
</threat_model>
